# E2E Tests for the Vercel Deployment

**Goal:** Catch regressions in the production deployment (`https://homefinder.patbuilds.dev`) before users do, with focus on the critical user journeys after the recent Celery → synchronous-scrape rewrite (commit `550b2bc`).

**Non-goals:** Replacing existing unit/integration tests in `apps/WebScraper/tests/`. Those still run via pytest; this plan adds a new outer ring that tests the deployed app over HTTP and (selectively) in a real browser.

---

## Why now

The architecture pivot (drop Celery, run scrape synchronously) collapsed two failure-prone subsystems into one request handler. That's mostly good — fewer moving parts — but it also means a single regression in the scrape path now takes down the user-facing flow with no async retry. Existing pytest suites mock most of that out. We need tests that prove the *deployed* artifact actually works end-to-end against real Vercel + real Neon.

---

## Tooling

**Choice: Playwright (Python) + pytest, layered with `httpx` for header-only checks.**

| Need | Tool | Why |
|---|---|---|
| Browser-driven UI flows (form submit, loading state, redirect) | Playwright | Already partial setup in `.playwright-mcp/`. Modern, headless, fast. |
| HTTP smoke checks (status codes, headers, content sniffing) | `httpx` + `pytest` | Don't spin up a browser to verify a 200. |
| Test runner / fixtures / parallelism | `pytest` + `pytest-playwright` | Matches the existing `pytest.ini` setup. |

Rejected:
- **Cypress** — would split the test ecosystem (Node + Python) and the existing harness is already pytest.
- **Selenium** — slower than Playwright with no upside here; the codebase removed its Selenium dependency from the request path already.
- **curl-only** — won't catch JS-rendered loading states or client-side filter behavior.

---

## Environments

| Env | URL | Use for |
|---|---|---|
| `prod` | `homefinder.patbuilds.dev` | Smoke tests only, **read-only**. No POSTs that mutate DB. Run on cron. |
| `preview` | Vercel preview URL per PR | Full E2E including write paths. Run on every PR. |
| `local` | `http://localhost:8000` | Dev iteration. Mock PCPAO via `responses` lib so tests don't hammer the real county site. |

The split matters: production has 67 real properties and a 60s/IP rate limit; we don't want CI runs polluting it or getting throttled. Preview deploys point at the same Neon DB though — that's a wart we'll have to live with unless we provision a separate test DB (see Open Questions).

## Test categories

### 1. Smoke (read-only, runs against `prod` on cron)

| # | Test | Assertion |
|---|---|---|
| S1 | `GET /` | 200, contains "Pinellas" |
| S2 | `GET /scraper/` | 200, form contains city options + property type checkboxes |
| S3 | `GET /scraper/dashboard/` | 200, ≥1 property card rendered |
| S4 | `GET /scraper/property/<known-parcel>/` | 200, address + market value visible |
| S5 | `GET /scraper/property/INVALID-PARCEL/` | 404 |
| S6 | `GET /scraper/download/excel/` | 200, `Content-Type: …spreadsheetml…`, valid xlsx magic bytes |
| S7 | `GET /scraper/download/pdf/` | 200, `Content-Type: application/pdf`, `%PDF-` header |
| S8 | `GET /admin/login/` | 200 |
| S9 | TLS + headers | HTTPS only, `Strict-Transport-Security` present, `X-Content-Type-Options: nosniff` |

These should run in under 30s total. Use a known-good parcel ID for S4 — fixture, refresh annually.

### 2. Functional (write paths, runs against `preview` on PR)

| # | Test | Setup | Assertion |
|---|---|---|---|
| F1 | Submit search with `city=Clearwater limit=2` | Fresh CSRF | 302 → dashboard with `?city=Clearwater`, returns within 30s, ≥1 new row in DB with `last_scraped` within last 60s |
| F2 | Submit two searches in rapid succession | — | First 302s; second renders the form with the rate-limit error message |
| F3 | Submit with `limit=999` | — | Caps at 25 (verify by counting `last_scraped` updates) |
| F4 | Submit with no filters | — | Still 302s, no 500 |
| F5 | Submit with bogus `city=NonExistentTown` | — | 302 to dashboard (empty result is OK), no 500 |
| F6 | Dashboard filter chain | After F1 ran | `?city=Clearwater&property_types=Single+Family` returns only matching cards |
| F7 | Dashboard pagination | DB has >12 rows | `?page=2` loads second page |

### 3. Browser flows (Playwright, runs on PR for `preview`)

| # | Test | Steps |
|---|---|---|
| B1 | Form submit shows loading state | Visit `/scraper/`, fill city, click submit, assert button has loading class within 100ms |
| B2 | Loading state clears on redirect | Continue from B1, wait for navigation, assert URL is `/scraper/dashboard/` |
| B3 | Property card click → detail page | Visit dashboard, click first card, assert URL changes to `/scraper/property/...` |
| B4 | Mobile breakpoint renders | 375px viewport, assert form fits without horizontal scroll |
| B5 | Filter sidebar toggles | Dashboard, click filter button, assert sidebar visible |

### 4. Performance budget (informational, not blocking)

- TTFB < 2s on `/scraper/`
- Scrape with `limit=5` finishes in < 30s end-to-end
- Lighthouse score ≥ 80 on `/`

These run nightly, surface as warnings, don't block deploys.

## Implementation phases

**Phase 1 — Smoke suite (≈2 hrs)**
- Create `tests/e2e/conftest.py` with `BASE_URL` fixture (env-driven, defaults to prod)
- Create `tests/e2e/test_smoke.py` with S1–S9
- Run manually against prod to baseline
- Add `make e2e-smoke` target

**Phase 2 — Functional suite (≈3 hrs)**
- `tests/e2e/test_functional.py` with F1–F7
- CSRF helper in `conftest.py` (fetch token, post with cookie jar)
- Fixture to query Neon directly for DB-side assertions
- `BASE_URL=preview-url make e2e-functional`

**Phase 3 — Browser suite (≈2 hrs)**
- `tests/e2e/browser/test_journeys.py` for B1–B5
- `playwright install chromium` in CI
- Headed mode locally (`--headed` flag) for debugging

**Phase 4 — CI wiring (≈1 hr)**
- Extend `.github/workflows/ci.yml` with new `e2e` job
- On PR: run smoke + functional + browser against preview deploy URL (extract from Vercel deployment status check)
- New nightly `.github/workflows/smoke.yml` on `cron: '0 6 * * *'` against prod
- Slack/email on smoke failure (optional)

**Phase 5 (optional) — Performance + visual regression**
- Lighthouse CI as a non-blocking job
- Playwright screenshot diff for landing pages

## File layout

```
tests/e2e/
  conftest.py            # BASE_URL, csrf_session, db_connection fixtures
  test_smoke.py          # S1–S9
  test_functional.py     # F1–F7
  browser/
    test_journeys.py     # B1–B5
    playwright.config.py
.github/workflows/
  ci.yml                 # extended with e2e job for PRs
  smoke.yml              # new, nightly cron against prod
Makefile                 # add e2e-smoke, e2e-functional, e2e-browser targets
```

`tests/e2e/` lives outside `apps/` to keep it separate from the existing pytest discovery (`testpaths = apps` in `pytest.ini`). The new tests will need their own pytest invocation, e.g. `pytest tests/e2e/`.

## Constraints and gotchas

- **PCPAO rate limit / flakiness.** Real PCPAO can throttle or 502. Mitigations: set `limit=1` or `limit=2` in CI; use a known-stable city (Clearwater); retry transient HTTP errors at the test layer with backoff.
- **Vercel function 60s timeout.** Caps `limit` at ~10 in tests to leave headroom. F3 verifies the in-app cap of 25 holds.
- **60s/IP rate limit.** Tests need to either run sequentially with sleeps, set unique `X-Forwarded-For` per test (the app trusts that header), or seed the cache table to bypass.
- **Shared Neon DB between prod and preview deploys.** Functional tests will leave rows in the production DB. Acceptable trade-off short-term; long-term, provision a second Neon project for previews and override `DATABASE_URL` per-deployment via Vercel env scoping.
- **CSRF.** Form posts need a fresh token from a prior GET. Helper in conftest handles this.
- **Cookie persistence.** Same `httpx.Client` reused across a test — rate-limit logic looks at `REMOTE_ADDR`, not cookies, so cookie state mostly doesn't matter for rate limiting.

---

## Open questions

1. **Test DB strategy.** Share Neon with prod (cheap, dirty), or provision a separate Neon project for preview deploys (clean, requires Vercel env scoping)? Recommend: start shared, migrate to separate when CI starts polluting visibly.
2. **How often should smoke run on prod?** Hourly is overkill, daily is too sparse. Recommend: every 4 hrs (`0 */4 * * *`).
3. **Block PRs on E2E failure?** Smoke and functional yes, browser tests yes, performance budget no.
4. **Preview deploy URL discovery.** Vercel writes the URL to a deployment status; need a small step to extract it for the test job. Either parse from `vercel deploy --prebuilt` output or use the Vercel GitHub Action.
5. **Test data cleanup.** Should F1–F5 delete the rows they create? Probably not — caching is by `last_scraped` and the rows are real PCPAO data. Cleanup adds complexity for marginal benefit.

---

## Success criteria

- Smoke suite runs green against prod, takes < 30s
- Functional suite catches a deliberately introduced regression (e.g. break the `redirect('dashboard')` line) on a PR before merge
- CI runs all three suites in < 5 min on a PR
- One smoke failure in production has paged us before a user reported it
